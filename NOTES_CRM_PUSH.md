# Notes → CRM Push

How the "Send to CRM" feature works — from config to wire format to HTML rendering.

---

## Overview

After a call note is generated by the notes agent, it can be pushed to the CRM via a single API call. Shinobi posts the note as a form-urlencoded HTTP request to the CRM's incoming endpoint. The CRM renders the note field as HTML using HTMLPurifier — **plain markdown is stripped and will appear as raw symbols**. Notes must be formatted as CRM-compatible HTML.

Tested against brtcrm.io on 2026-04-28.

---

## Enabling It

The feature is **disabled by default**. Set these in your `.env` file on the VM:

```env
CRM_PUSH_ENABLED=true
CRM_PUSH_API_USERNAME=your_crm_username
CRM_PUSH_API_PASSWORD=your_crm_password
CRM_PUSH_API_KEY=your_crm_api_key

# Optional — defaults shown
CRM_PUSH_ENDPOINT=https://brtcrm.io/api/v1/accounts/{api_username}-incoming/
CRM_PUSH_DATA_FIELD=note
CRM_PUSH_TIMEOUT_S=20
```

If `CRM_PUSH_ENABLED` is not `true`, the endpoint returns HTTP 403.

---

## Triggering a Push

**Endpoint:** `POST /notes/{note_id}/send-to-crm`

```json
{ "account_id": "111111" }
```

`account_id` is optional — if omitted, Shinobi resolves it from the CRMPair table using the note's agent + customer. Returns HTTP 400 if it can't resolve one.

---

## How the CRM Endpoint Is Resolved

Shinobi builds candidate endpoints in order and tries each until one returns HTTP < 400:

1. **Configured endpoint** (`CRM_PUSH_ENDPOINT`) with substitutions:
   - `{api_username}` → value of `CRM_PUSH_API_USERNAME`
   - `{crm_base}` → base URL from CRMPair's `crm_url` (e.g. `https://brtcrm.io`)
   - `{crm_host}` → hostname only
   - If path contains `/api/v1/accounts/` but not `-incoming`, it is auto-corrected to `…/{api_username}-incoming`

2. **Auto-derived endpoint** from the CRMPair's stored `crm_url`:
   ```
   {crm_base}/api/v1/accounts/{api_username}-incoming
   ```

First endpoint that returns 2xx wins. If all fail, returns HTTP 502 with the full attempt log.

## What Gets Sent to the CRM

### HTTP Request

```
POST https://brtcrm.io/api/v1/accounts/{username}-incoming
Content-Type: application/x-www-form-urlencoded
```

### Form Fields

| Field | Value |
|---|---|
| `api_username` | `CRM_PUSH_API_USERNAME` env var |
| `api_password` | `CRM_PUSH_API_PASSWORD` env var |
| `api_key` | `CRM_PUSH_API_KEY` env var |
| `account_id` | CRM account ID for the customer |
| `data` | JSON string — see below |

### The `data` Field (JSON string)

```json
{
  "note": "<HTML-formatted note content>",
  "call_id": "123456",
  "agent": "Adam B Carter",
  "customer": "Danny Powell",
  "model": "gpt-5.4",
  "created_at": "2026-04-30T09:27:05.123456"
}
```

The key name `"note"` is controlled by `CRM_PUSH_DATA_FIELD`.

## CRM HTML Formatting

The CRM uses HTMLPurifier. Only a specific whitelist of tags and CSS properties renders — everything else is stripped.

### Allowed Tags

| Tag | Effect |
|---|---|
| `<strong>` | Bold |
| `<b>` | Bold (same as strong) |
| `<u>` | Underline |
| `<em>` | Italic |

### Allowed CSS (via `style=` attribute)

| Property | Example |
|---|---|
| `font-size` | `style="font-size:18px"` |
| `color` | `style="color:red"` or `style="color:#cc0000"` |
| `font-style` | `style="font-style:italic"` |
| `text-decoration` | `style="text-decoration:line-through"` |
| `background-color` | `style="background-color:yellow"` |

### Other Rendering

| Syntax | Renders as |
|---|---|
| Actual newline `\n` | Line break |
| `&nbsp;` | Indentation / spacing |
| Unicode, emoji | Render natively (✅ ⚠️ ●  ◦ ★) |

## Markdown → CRM HTML Mapping

The notes agent must output HTML using this mapping — raw markdown will not render.

| Intent | CRM HTML to use |
|---|---|
| H1 heading | `<strong style="font-size:22px">text</strong>` |
| H2 heading | `<strong style="font-size:18px">text</strong>` |
| H3 heading | `<strong style="font-size:15px">text</strong>` |
| Bold | `<strong>text</strong>` |
| Italic | `<em>text</em>` |
| Underline | `<u>text</u>` |
| Strikethrough | `<strong style="text-decoration:line-through">text</strong>` |
| Bold + italic | `<strong><em>text</em></strong>` |
| Bold + italic + underline | `<strong><em><u>text</u></em></strong>` |
| Bullet item | `● item` |
| Nested bullet | `&nbsp;&nbsp;◦ item` |
| Numbered item | `1. item` (manual numbering) |
| Done / checked | `✓ item` |
| Pending | `◻ item` |
| Important | `★ item` |
| Horizontal rule | `━━━━━━━━━━━━━━━━━━━━━━━━━━━━` |
| Blockquote | `<em style="color:gray">&nbsp;&nbsp;❝ text</em>` |
| Inline code | `<u style="font-size:12px">text</u>` |
| Table row | `<strong>Key:</strong> Value` (one per line) |

### Callout Boxes

```html
<strong style="color:green">✅ Success — action completed</strong>
<strong style="color:red">❌ Error — something failed</strong>
<strong style="color:orange">⚠️ Warning — needs attention</strong>
<strong style="color:blue">ℹ️ Info — note for reference</strong>
```

## What Does NOT Work

The following are stripped entirely by HTMLPurifier:

- `<h1>`–`<h3>` heading tags
- `<ul>`, `<ol>`, `<li>` list tags
- `<br>` line break tag (use actual `\n` instead)
- `<s>`, `<del>`, `<strike>` strikethrough tags
- `<sup>`, `<sub>` super/subscript
- `<span>`, `<div>`, `<p>` container tags
- `<font>` tag
- `<blockquote>`, `<code>`, `<mark>`
- `<details>`, `<summary>` collapsible
- Markdown syntax (`#`, `**`, `-`, `>`, `` ` ``) — renders as literal characters
- `font-family: monospace` — stripped (no fixed-width columns)
- Tab characters `\t` — no column alignment
- Inline styles on `<span>`, `<div>`, `<p>` — entire tag stripped

This is the structure notes should use when pushed to the CRM:

```html
<strong style="font-size:22px">📋 Call Summary</strong>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
<strong>Agent:</strong> Adam B Carter
<strong>Customer:</strong> Danny Powell
<strong>Date:</strong> 2026-04-28
<strong>Duration:</strong> 12m35s

<strong style="font-size:18px">Summary</strong>
● What was discussed and key outcomes
● Customer's stance at end of call
&nbsp;&nbsp;◦ Sub-point if needed

<strong style="font-size:18px">Sales Techniques Used</strong>
● Specific tactic observed
● Objection handled: <em>price concern addressed with ROI framing</em>

<strong style="font-size:18px">Compliance & Risk</strong>
<strong style="color:orange">⚠️ Risk: Medium</strong>
● Required disclosure given: ✓

<strong style="font-size:18px">Communication Quality</strong>
● Tone: professional

<strong style="font-size:18px">Next Steps</strong>
✓ Follow-up email sent
◻ Schedule second call by Friday
★ <strong>Send product comparison doc</strong>

<em style="color:gray;font-size:12px">Generated by Shinobi · gpt-5.4</em>
```

## API Response

On success:

```json
{
  "ok": true,
  "message": "Note sent to CRM",
  "crm_status": 200,
  "endpoint": "https://brtcrm.io/api/v1/accounts/myuser-incoming",
  "account_id": "111111",
  "note_id": "uuid-here",
  "crm_response": "...",
  "attempts": []
}
```

On failure (all endpoints rejected):

```json
{
  "detail": {
    "message": "CRM push failed on all candidate endpoints",
    "attempts": [
      { "request": { "endpoint": "..." }, "status": 404, "response": { ... } }
    ],
    "note_id": "uuid-here",
    "account_id": "111111"
  }
}
```

## Logs

Every push is printed to the backend log (credentials masked):

```
[crm-push] note=<id> request={"endpoint": "...", "body_form": {"api_password": "abc***xyz", ...}}
[crm-push] note=<id> response={"status": 200, "body_preview": "..."}
```
